# md5__1038 原理与会话预热说明

## 1. 先说结论

1. `md5__1038` 名字里有 `md5`，但它**不是普通的静态 MD5**，而是由页面下发的 WAF 挑战脚本在运行时生成的动态签名参数。  
2. 对部分接口，是否能访问不只取决于 `md5__1038`，还取决于会话上下文（匿名 token/cookie 是否齐全）。  
3. `https://xueqiu.com/` 与 `https://xueqiu.com/hq` 的“预热效果”不同：  
   - 只打开 `/`：通常只拿到 `acw_tc`，可能仍报 `400016`。  
   - 先打开 `/hq`：会补齐 `xq_a_token/xqat/xq_r_token/xq_id_token/cookiesu/u` 等匿名会话 cookie，然后 API 才能稳定返回 JSON。  

---

## 2. `md5__1038` 是怎么来的（当前实现视角）

当前项目不是“手工还原某个固定公式”，而是采用“运行挑战脚本并抓结果”的方式：

1. 先请求目标 API（去掉旧 `md5__1038`）；
2. 如果返回的是挑战页（HTML 中包含 `aliyunwaf_`、`renderData` 等），提取挑战脚本；
3. 在本地 Node 运行时（`tmp_run_waf3.js`）中执行挑战脚本；
4. 监听脚本对 URL 的改写（`anchor.href`/history/fetch/XHR 等），抓到带 `md5__1038` 的最终 URL；
5. 把该 URL 中的 `md5__1038` 回填到原始请求参数里。

这也是为什么 `md5__1038` 每次看起来都很长、而且会变化：它依赖当次挑战上下文，而不是固定明文公式。

---

## 3. “首次必须打开首页，第二次 API 才能用”本质原因

你遇到的现象是典型会话校验：

1. 直接打 API（哪怕带了 `md5__1038`）会返回：
   - `{"error_code":"400016", ...}`  
2. 先访问 `https://xueqiu.com/` 再打 API，仍可能 `400016`；  
3. 先访问 `https://xueqiu.com/hq` 再打 API，返回 `200 application/json`。

根因是 `/hq` 会下发后续接口校验所需的匿名会话 cookie，而 `/` 不完整。

---

## 4. 这次如何解决

`xueqiu_md5_flow.py` 里已做以下处理：

1. **固定先预热 `/hq`**（不是 `/`）；  
2. 请求前自动去掉旧 `md5__1038`，避免旧签名干扰；  
3. 新增“前置判断”：如果 unsigned URL 本身已返回 JSON，直接跳过 signer；  
4. 若返回挑战页，再走 signer 动态生成；  
5. 把 `error_code`（如 `400016`）识别为失败，而不是“JSON 成功”；  
6. 成功后自动把最终 JSON 写入本地 `xueqiu_response.json`。

---

## 5. 现在的判定标准

执行：

```bash
python xueqiu_md5_flow.py
```

满足以下两点即判定成功：

1. `verify ... json: True`；  
2. 本地生成 `xueqiu_response.json`（文件内容是接口返回数据，不是挑战页/错误 JSON）。

---

## 6. 实战建议

1. 若出现 `400016`，先检查是否做了 `/hq` 预热；  
2. 不要长期复用旧 `md5__1038`，优先“实时生成或直接走 unsigned（若可）”；  
3. 先区分两类失败：  
   - `text/html`（挑战页）；  
   - `application/json + error_code`（会话校验失败）；  
4. 对不同接口分别验证，不要假设同一接口策略永久不变。
